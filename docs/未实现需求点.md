# 未实现需求点

## 记录时间

2026-07-06

## 背景

当前扩展主要面向 X 网页端的时间线帖子：自动展开被折叠的帖子内容，调用 X Web 自带的 Grok 翻译接口，并将时间线中识别到的帖子正文替换为中文译文。

目前还有一类重要场景没有覆盖：X 的文章、帖子详情页和评论区内容也需要自动翻译。这个文档先作为待讨论需求记录，不在本次改动中展开方案设计或实现。

## 需求摘要

后续需要支持对 X 页面中更多内容形态进行自动翻译，包括但不限于：

- X 的长文、文章类内容。
- 单条帖子详情页中的主帖正文。
- 帖子详情页下方的评论、回复和回复树。
- 动态加载出来的后续评论内容。
- 被折叠、需要展开后才能看到完整文本的评论或长内容。

目标是让用户在浏览 X 时，不仅时间线帖子可以自动翻译，进入文章、帖子详情或查看评论时，也能尽量自动看到中文译文。

## 当前未实现范围

### 文章和长文

- 可能存在与普通时间线帖子不同的 DOM 结构。
- 正文长度可能明显长于普通帖子。
- 内容可能被拆分成多个段落、链接、引用或富文本节点。
- 需要确认 X 自带翻译接口是否支持同类内容，或者是否仍然可以通过帖子 ID 获取译文。

### 帖子详情页

- 当前逻辑倾向于处理时间线中的帖子列表。
- 进入单条帖子详情页后，主帖、引用帖、相关推荐帖和评论可能混在同一页面中。
- 后续需要明确哪些内容应自动翻译，哪些内容应跳过，避免重复翻译或误翻译。

### 评论和回复

- 评论区通常随滚动持续加载。
- 回复树层级可能较深，同一条评论可能出现在不同上下文中。
- 需要避免同一评论反复请求翻译。
- 需要保持页面滚动和交互体验稳定，不能因为翻译插入或替换导致明显跳动。

## 后续讨论点

- 如何识别文章、主帖、评论、回复、引用帖等不同内容类型。
- 是否继续只使用 X 自带翻译能力，避免引入第三方翻译服务。
- 翻译缓存是否沿用当前帖子 ID 缓存策略，还是需要增加内容类型、页面位置或上下文信息。
- 详情页是否应该翻译所有可见评论，还是只翻译视口附近内容。
- 已翻译内容是否仍然只替换正文，还是需要提供查看原文的方式。
- 对失败、空译文、接口限流等情况如何展示状态。
- 是否需要在插件弹窗中增加文章和评论相关的诊断计数。

## 初步验收方向

后续实现完成后，可以按以下方向验证：

- 打开 X 时间线，现有帖子自动翻译能力不回退。
- 打开单条帖子详情页，主帖正文可以自动翻译。
- 向下滚动评论区，新加载的评论可以自动翻译。
- 展开被折叠的长评论或长文后，可以获取完整译文。
- 同一条内容重复出现时，不会频繁重复请求翻译。
- 翻译失败时页面仍可正常浏览，并能看到简短状态提示或诊断信息。
