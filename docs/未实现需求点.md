# 未实现需求点

## 记录时间

2026-07-06

## 背景

当前扩展主要面向 X 网页端的时间线帖子：自动展开被折叠的帖子内容，调用 X Web 自带的 Grok 翻译接口，并将时间线中识别到的帖子正文替换为中文译文。

目前曾有一类重要场景没有覆盖：X 的文章、帖子详情页和评论区内容也需要自动翻译。随着详情页评论识别和第三方翻译服务接入推进，普通详情页主帖、评论、回复以及 X Article 长文已经进入实现阶段；本文继续记录已完成范围、当前限制和后续待讨论点。

## 需求摘要

需要支持对 X 页面中更多内容形态进行自动翻译，包括但不限于：

- X 的长文、文章类内容：已通过第三方 provider 管线支持，当前可用 provider 为腾讯翻译。
- 单条帖子详情页中的主帖正文：已支持。
- 帖子详情页下方的评论、回复和回复树：已支持可见 article 的自动识别与翻译。
- 动态加载出来的后续评论内容：已通过页面观察逻辑调度处理。
- 被折叠、需要展开后才能看到完整文本的评论或长内容。

目标是让用户在浏览 X 时，不仅时间线帖子可以自动翻译，进入文章、帖子详情或查看评论时，也能尽量自动看到中文译文。

## 当前实现状态

### 文章和长文

- 已识别 X Article 的主长文容器：`[data-testid="twitterArticleReadView"]`。
- 已提取标题：`[data-testid="twitter-article-title"]`。
- 已提取正文根：`[data-testid="longformRichTextComponent"]`。
- 已避免把引用卡片、`role="link"` 卡片和 `card.wrapper` 中的长文误当成主长文。
- 长文不再调用 X 自带 `grok/translation.json` 帖子译文接口，因为该接口不稳定返回整篇长文译文。
- 长文直接进入第三方 provider 翻译管线；当前已接入腾讯翻译，后续可以继续扩展更多 provider。
- 长文译文以独立译文块渲染在文章阅读区域后方，保留 X 原始富文本 DOM，避免破坏链接、媒体或页面交互。
- 未配置第三方翻译服务时，页面显示“配置第三方翻译服务后可翻译长文”。

### 帖子详情页

- 详情页 `https://x.com/:user/status/:id` 已允许处理。
- 主帖、评论和回复统一按 `article[data-testid="tweet"]` 识别。
- 每个 article 只选择自己的主 `[data-testid="tweetText"]`，避免翻译引用卡片内容。
- 引用帖、推荐卡片和卡片内嵌正文会被排除，减少误翻译。

### 评论和回复

- 评论区随滚动持续加载时，新增 article 会被观察并调度翻译。
- 同一条内容按帖子 ID 使用缓存和 in-flight 去重，避免频繁重复请求。
- 普通评论仍优先使用 X 自带翻译，只有限流、网络异常、JSON 解析异常或 5xx 等可兜底失败时才进入第三方 provider。
- 页面滚动和交互仍需在真实 X 页面持续观察，重点关注长文译文块插入后的视觉稳定性。

## 当前仍有限制

- `https://x.com/i/article/:id` 直达文章页仍不支持，当前处理范围主要是 status 页面内的 X Article。
- 长文当前依赖第三方 provider；如果未配置 provider，就不会伪造译文或回退到不完整的 X 自带接口结果。
- 腾讯翻译单次文本长度存在 provider 限制，超长文章可能需要后续按段落切分、分批翻译和合并渲染。
- 当前第三方 provider 只有腾讯翻译，未来接入更多厂商时应继续复用统一 provider 管线，不把长文逻辑绑定到某个厂商。
- 已翻译长文目前保留原文并追加译文块；是否提供原文/译文切换、复制、重新翻译等交互，后续再讨论。

## 后续讨论点

- 如何识别文章、主帖、评论、回复、引用帖等不同内容类型。
- 第三方 provider 的选择、排序、限额耗尽后的降级策略。
- 翻译缓存是否沿用当前帖子 ID 缓存策略，还是需要增加内容类型、页面位置或上下文信息。
- 详情页是否应该翻译所有可见评论，还是只翻译视口附近内容。
- 已翻译内容是否仍然只替换正文，还是需要提供查看原文的方式。
- 对失败、空译文、接口限流、第三方 provider 未配置或额度耗尽等情况如何展示状态。
- 是否需要在插件弹窗中增加文章和评论相关的诊断计数。
- 超长文章是否按段落切分翻译，以及如何保证段落顺序、缓存粒度和重试体验。

## 初步验收方向

后续实现完成后，可以按以下方向验证：

- 打开 X 时间线，现有帖子自动翻译能力不回退。
- 打开单条帖子详情页，主帖正文可以自动翻译。
- 向下滚动评论区，新加载的评论可以自动翻译。
- 打开包含 X Article 的 status 页，配置第三方 provider 后长文可以显示中文译文。
- 未配置第三方 provider 时，长文区域显示明确提示，不调用 X 自带帖子译文接口。
- 展开被折叠的长评论或长文后，可以获取完整译文。
- 同一条内容重复出现时，不会频繁重复请求翻译。
- 翻译失败时页面仍可正常浏览，并能看到简短状态提示或诊断信息。

## 当前状态

已实现详情页主帖、评论、动态评论和 X Article 长文的基础自动翻译能力。长文采用第三方 provider 管线，当前可通过腾讯翻译完成；剩余工作主要是超长文章分段、多 provider 策略、直达文章页支持和更细的交互体验。
